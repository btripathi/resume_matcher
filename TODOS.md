## HIGH PRIORITY

### 1. Add threading.RLock guard around runtime_settings (ARCH-1)
**What:** Wrap all reads/writes to `runtime_settings` dict in `backend/app.py` with a `threading.RLock`.
**Why:** The dict is shared across HTTP threads and JobRunner's thread pool with no synchronization. Concurrent read/write can cause intermittent LLM config corruption.
**Where to start:** `backend/app.py` lines 71-86 (dict definition), then grep for all `runtime_settings[` accesses.
**Effort:** ~20 lines changed. CC: ~5min.

### 2. Atomic deep scan checkpoint writes (ARCH-2)
**What:** Wrap checkpoint writes in `job_runner.py` (`deep_resume_from`, `deep_partial_details`, `result_json`, `progress`) in a single `BEGIN/COMMIT` DB transaction.
**Why:** If the server crashes between separate writes, the durable queue resumes from inconsistent state — defeating its purpose.
**Where to start:** `backend/services/job_runner.py` checkpoint logic (~line 744). May need a new `database.py` method that accepts multiple updates in one transaction.
**Effort:** ~10 lines + small DB helper. CC: ~5min.

### 3. Daemon threads + Event shutdown for heartbeat (ARCH-3)
**What:** Mark heartbeat threads as `daemon=True` and use `threading.Event` for cooperative shutdown instead of the current 1s join timeout.
**Why:** Blocked heartbeat threads become orphaned, leaking resources. Compounds over 50+ runs.
**Where to start:** `backend/services/job_runner.py` heartbeat spawn (~line 79) and shutdown (~line 146).
**Effort:** ~15 lines. CC: ~5min.

### 4. Enable SQLite WAL mode (ARCH-4)
**What:** Add `conn.execute('PRAGMA journal_mode=WAL')` to `database.py` connection setup.
**Why:** Without WAL, readers block writers. With 16 concurrent job runner threads, this causes lock contention and 30s timeouts.
**Where to start:** `database.py` `get_connection()` function (line 11).
**Effort:** 1 line. CC: ~2min.

### 5. Full pytest test suite (TEST-1)
**What:** Add pytest with ~40 tests covering: ai_engine mock paths, JSON parse + fallback recovery, database CRUD + migrations, job_runner state machine (queued->running->done/failed/paused), deep scan matching passes, document extraction (PDF/DOCX fixtures), endpoint integration via FastAPI TestClient, concurrent queue stress test.
**Why:** Zero automated tests exist. The codebase has complex retry logic, a 4-pass matching algorithm, a durable job queue, and concurrent thread pools. ai_engine mock mode makes this feasible without an LLM.
**Where to start:** Create `tests/` directory. Add `pytest` to requirements.txt. Start with `test_database.py` (CRUD) and `test_ai_engine.py` (mock mode), then build up.
**Depends on:** WAL mode (TODO-4) should be enabled first to avoid test flakiness from lock contention.
**Effort:** human: ~1 week / CC: ~1 hour.
